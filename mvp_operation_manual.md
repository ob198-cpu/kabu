# 株式購入判断補助システム MVP 運用手順書

作成日: 2026年5月14日

## 1. 目的

本手順書は、MVP版の株式購入判断補助システムを毎日更新し、予測・実績・誤差を記録するための運用方法を定める。

このシステムは投資助言、利益保証、自動売買を行わない。実売買前には、証券会社画面、企業IR、最新決算、市場イベントを別途確認する。

## 2. 毎日行う作業

| 順番 | 作業 | 内容 |
|---:|---|---|
| 1 | データ更新 | 株価、出来高、指数、為替、金利、PER/PBR/EPSの一部を取得 |
| 2 | スコア計算 | トレンド、モメンタム、出来高、割安性、市場、財務、リスクを計算 |
| 3 | 購入前チェック | VIX、米10年金利、日経平均、NASDAQ、イベント通過状況を確認 |
| 4 | 予測ログ保存 | 当日のスコアと判断を `mvp_predictions.csv` に保存 |
| 5 | 実績確認 | 検証日が来た予測について、実績リターンを `mvp_prediction_actuals.csv` に保存 |
| 6 | ブック確認 | `stock_decision_mvp_calculated.xlsx` の各シートを確認 |

## 3. 実行コマンド

PowerShellで以下を実行する。

```powershell
cd C:\AI関連ファイル\株
powershell.exe -ExecutionPolicy Bypass -File tools\run_daily_mvp_update.ps1
```

## 4. 出力ファイル

| ファイル | 内容 |
|---|---|
| `reports\stock_decision_mvp_calculated.xlsx` | 実データ、スコア、チェックリスト、履歴を含むMVPブック |
| `reports\stock_decision_mvp_data.json` | 取得データとスコアのJSON |
| `reports\mvp_score_history.csv` | 日次スコア履歴 |
| `reports\mvp_predictions.csv` | 予測ログ |
| `reports\mvp_prediction_actuals.csv` | 予測と実績の比較 |

## 5. 確認するシート

| シート | 確認内容 |
|---|---|
| `07_scores` | 当日の機械スコアと判定 |
| `12_pre_buy_checklist` | 購入停止条件が出ていないか |

| `15_review_adjusted` | 決算確認後の補正スコア |

| `09_errors` | 予測と実績の比較状況 |

## 6. 判定ルール

| 条件 | 行動 |
|---|---|
| スコア80点以上 | 買い候補。ただし決算・イベント確認後 |
| スコア70〜79点 | 少額候補。ただし決算・イベント確認後 |
| スコア60〜69点 | 様子見 |
| スコア59点以下 | 除外/待機 |
| VIX 30以上 | 新規購入停止 |
| 米10年金利 4.70%以上 | 高PER銘柄を停止 |
| 日経平均またはNASDAQが急落 | 新規購入停止または延期 |
| CPI/FOMC/日銀前 | 原則購入しない |
| 決算で下方修正 | 当該銘柄を除外 |

## 7. 6月の運用予定

| 日程 | 作業 |
|---|---|
| 6/1〜6/9 | 毎日更新、仮想売買、誤差記録。実購入なし |
| 6/10 | 米CPI確認 |
| 6/16〜6/17 | FOMC確認 |
| 6/19 | 日銀会合確認 |
| 6/22〜6/26 | 条件を満たす場合のみNISA少額テスト |
| 6/29〜6/30 | 初回レビュー。追加購入は慎重判断 |

## 8. 購入テスト上限

| 想定資金 | 初回購入テスト上限 |
|---:|---:|
| 100万円 | 5万〜10万円 |
| 200万円 | 10万〜20万円 |

## 9. 異常時の対応

| 異常 | 対応 |
|---|---|
| データ取得失敗 | 当日のスコアを購入判断に使わない |
| PER/PBR/EPSが未取得 | 公式IR、四季報、証券会社画面で手動確認 |
| スコアが急変 | 価格データ、分割、異常値、出来高を確認 |
| 予測と実績が大きく外れた | `09_errors` に理由分類を記録 |
| GitHub Pagesが古い | ローカルファイルを優先確認。反映待ちの可能性あり |

## 10. 次工程

1. 実績比較の自動精度を上げる
2. 決算データの手入力フォームを整備する
3. 売上成長率、営業利益成長率、下方修正判定をスコア化する
4. 6月上旬から毎日運用し、誤差を蓄積する
5. 6月イベント通過後に少額NISAテスト可否を判断する
